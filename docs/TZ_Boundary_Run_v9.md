# ТЕХНИЧЕСКОЕ ЗАДАНИЕ
## Boundary Run: The Little Signal — Расширение v9.0
### Три направления: Геймплей | Нарратив | Техническая платформа

---

## 0. КОНТЕКСТ И ОГРАНИЧЕНИЯ

**Базовая архитектура:**
- Zero-backend, zero-telemetry, zero-analytics
- Deterministic replay proof (Rust core → WASM → Canvas/WebGL)
- Single-page browser application, no install
- Apache-2.0 OR MIT license
- Educational simulation, not a medical device

**Нарушать запрещено:**
- Любые сетевые запросы (кроме первичной загрузки assets)
- localStorage/cookies для персональных данных
- Случайность без детерминированного seed
- Состояние, которое нельзя воспроизвести через replay proof

## НАПРАВЛЕНИЕ 1: ГЕЙМПЛЕЙНЫЕ МЕХАНИКИ
### Цель: повысить глубину принятия решений с сохранением образовательной метафоры

---

### 1.1 Система «Контрактов доставки» (Delivery Contracts)

**Описание:**
Перед каждым забегом игрок выбирает 0–2 необязательных условия, которые модифицируют правила и дают множитель к Proof Shards.

**Типы контрактов:**
| ID | Название | Условие | Множитель | Образовательная метафора |
|---|---|---|---|---|
| C-01 | «Zero Trust» | Нельзя использовать Revoke (отзыв согласия) | ×1.4 | Однажды данное согласие необратимо |
| C-02 | «Minimal Surface» | Нельзя пересекать центральную полосу более 3 раз | ×1.3 | Чем меньше attack surface, тем лучше |
| C-03 | «No Throttle» | Запрещён Throttle (торможение стимуляции) | ×1.5 | Полный запрет на вмешательство = высокий риск |
| C-04 | «Full Audit» | Нельзя пропустить ни одного Gate Audit | ×1.6 | Все запросы на доступ должны быть проверены |
| C-05 | «Sealed Envelope» | Нельзя пропустить ни одного Seal | ×1.5 | Целостность данных — абсолютный приоритет |

**Технические требования:**
- Контракты — часть `RunConfig`, сериализуются в replay proof
- Валидация выполнения контрактов — в runtime, без дополнительных вычислений
- UI: модальное окно перед `start_run()`, отключается через флаг `skip_contracts`

**Сложность реализации:** Low (2–3 дня)
**Приоритет:** P1 (высокий)

---

### 1.2 Моральные развилки (Path Branching)

**Описание:**
На дистанции появляются развилки — два параллельных коридора с разной плотностью угроз и разной наградой.

**Типы путей:**
- **Fast Lane:** короче на 20%, но утечки (signal leaks) в 2× чаще. Даёт больше Shards, но меньше Integrity Score.
- **Safe Lane:** длиннее на 30%, но есть Recovery Stations (восстанавливают «отозванное согласие»). Даёт меньше Shards, но Integrity → 1.0.
- **Audit Lane:** средняя длина, но каждые 5 секунд — обязательный Gate Audit. Пропуск = мгновенный Quarantine (не game over, но потеря 50% скорости на 3 сек).

**Образовательная метафора:**
- Fast Lane = convenience over privacy (торговля скоростью за безопасность)
- Safe Lane = privacy-by-design (медленнее, но надёжнее)
- Audit Lane = zero-trust architecture (постоянная верификация)

**Технические требования:**
- Развилки генерируются детерминированно от seed на позициях кратных 500 единицам дистанции
- Выбор пути записывается в replay proof как `BranchEvent { distance, choice }`
- Integrity Score — новое поле в `RunResult`, влияет на финальный replay proof hash

**Сложность реализации:** Medium (5–7 дней)
**Приоритет:** P1

---

### 1.3 Динамические «Нейронные погодные условия» (Neural Weather)

**Описание:**
Seed детерминированно выбирает 0–2 модификатора забега, которые меняют правила на всей дистанции.

**Модификаторы:**
| ID | Название | Эффект | Образовательная метафора |
|---|---|---|---|
| W-01 | Phishing Storm | 30% Gate Audit — ловушки (поддельные запросы). Неверный аудит = -10 Integrity. | Фишинг: не все запросы на согласие легитимны |
| W-02 | Boundary Erosion | Полосы медленно сужаются (от 3 до 2 за 60 сек). | Приватные границы эродируют без активной защиты |
| W-03 | Resonance Feedback | Каждый Throttle увеличивает частоту утечек на 15% на 10 сек. | Агрессивное вмешательство провоцирует обратную реакцию |
| W-04 | Stale Consent | Каждые 20 сек одно согласие «устаревает» (требует Re-audit). | Согласие — не вечно, требует переаудита |
| W-05 | Data Gravity | Seal появляются реже, но дают ×3 Shards. | Редкие, но ценные данные требуют большей защиты |

**Технические требования:**
- Модификаторы — часть `RunConfig`, вычисляются от seed через хеш-функцию
- Все эффекты — чистые функции от `(seed, distance, time)`
- UI-индикатор погоды — в HUD, обновляется каждые 100 единиц дистанции

**Сложность реализации:** Medium (4–6 дней)
**Приоритет:** P2

### 1.4 Система репутации курьера (Courier Reputation)

**Описание:**
Прогрессия без gameplay-влияния. Только косметика и QoL.

**Уровни:**
| Уровень | Shards (накопительно) | Награда |
|---|---|---|
| Neophyte | 0 | Базовый скин Ari |
| Courier | 500 | Скин «Encrypted» (синий glow) |
| Sentinel | 1500 | Мини-радар (показывает утечку за 0.4 сек раньше) |
| Archon | 3000 | Скин «Deterministic» (золотой, с trail-эффектом) |
| Sovereign | 5000 | Возможность «подписывать» replay proof своим псевдонимом (хеш от введённой строки + proof) |

**Технические требования:**
- Прогрессия хранится в `sessionStorage` (не persist между сессиями, zero-telemetry)
- Radar — визуальный hint, не изменяет hitbox или логику
- Подпись proof — клиентский хеш, не отправляется никуда

**Сложность реализации:** Low (2–3 дня)
**Приоритет:** P2

## НАПРАВЛЕНИЕ 2: ВИЗУАЛЬНАЯ ИСТОРИЯ / НАРАТИВ
### Цель: эмоциональное погружение без cutscenes и текста

---

### 2.1 Flashback-кадры при аудите (Memory Fragments)

**Описание:**
При успешном Gate Audit на 1–2 кадра (60–120 мс) показывается абстрактный образ — силуэт, геометрическая фигура, или цветовая вспышка, ассоциирующаяся с «человеком, чей выбор несёт Ari».

**Реализация:**
- 8–12 уникальных «фрагментов» — SVG-спрайты, 64×64 px, монохромные
- Появление: `opacity 0 → 1 → 0` за 100 мс, позиция — центр ворот
- Фрагмент выбирается детерминированно: `fragment_id = hash(seed + audit_index) % fragment_count`
- Без текста, без диалогов. Только визуальный якорь.

**Образовательная метафора:**
За каждым запросом на доступ стоит реальный человек. Аудит — это не бюрократия, это защита конкретного выбора.

**Технические требования:**
- SVG inline, не внешние запросы
- Анимация через CSS transition, не JS (производительность)
- Не блокирует gameplay, overlay поверх canvas

**Сложность реализации:** Low (1–2 дня)
**Приоритет:** P1

---

### 2.2 «Глитч-эффект» при ошибках (Data Corruption)

**Описание:**
Если игрок пропускает Seal или попадает в ловушку — интерфейс на 1–2 секунды «глючит»: текст заменяется на ████, canvas смещается на 2–3 px, цвет инвертируется.

**Реализация:**
- CSS-фильтры: `filter: invert(0.1) hue-rotate(90deg)` на 200 мс
- Текстовые элементы: замена символов на U+2588 через CSS `content` или JS (только UI, не canvas)
- Canvas: смещение render target на `sin(time * 50) * 3` px на 1 сек
- Всё — клиентское, никаких внешних assets

**Образовательная метафора:**
Пропущенная печать данных = повреждение. Последствия ошибок накапливаются и визуально ощутимы.

**Технические требования:**
- Эффект — чисто визуальный, не влияет на hitbox или логику
- Длительность и интенсивность зависят от `corruption_level` (0–3)
- `corruption_level` записывается в replay proof

**Сложность реализации:** Low (1–2 дня)
**Приоритет:** P2

---

### 2.3 Эволюция Ari (Visual Evolution)

**Описание:**
Внешний вид Ari меняется в зависимости от стиля прохождения — не по уровню, а по *поведению* в текущем забеге.

**Состояния:**
| Условие | Визуал | Метафора |
|---|---|---|
| Integrity > 0.9 | Ari окружён щитом-гало | Данные целы, граница нетронута |
| Много Throttle | Ari «трясётся», trail короткий | Агрессивное вмешательство истощает |
| Много Revoke | Ari становится полупрозрачным | Постоянный отзыв согласия размывает идентичность |
| Fast Lane + успех | Ari оставляет красный след | Скорость ценой риска |
| Safe Lane + успех | Ari оставляет синий след | Стабильность и предсказуемость |

**Технические требования:**
- Состояния вычисляются в runtime от `RunState`, не от истории
- Trail — частicle system на canvas, 20–30 частиц, zero overhead
- Переходы между состояниями — линейная интерполяция за 0.5 сек

**Сложность реализации:** Medium (3–4 дня)
**Приоритет:** P2

---

### 2.4 Финальный экран: «Чей выбор ты нёс?»

**Описание:**
После deterministic replay proof — не просто таблица очков, а абстрактная визуализация: на основе `Integrity Score` и выбранных путей генерируется уникальный «портрет» (процедурная геометрия).

**Реализация:**
- Процедурная генерация SVG: 3–5 пересекающихся кругов/линий, цвета от HSL(Integrity × 120°, 70%, 50%)
- Портрет детерминированен: `portrait_seed = hash(run_proof_hash)`
- Можно скачать как SVG (data URI, не сервер)
- Подпись: «Proof #<hash_prefix> | Integrity: <integrity> | Path: <path_signature>»

**Образовательная метафора:**
Каждое прохождение — уникальный след в цифровом пространстве. Но он детерминирован: если повторить те же действия — получишь тот же портрет.

**Технические требования:**
- SVG generation — чистый JS, no libraries
- Размер < 2 KB
- Скачивание через `Blob` + `URL.createObjectURL`

**Сложность реализации:** Medium (3–4 дня)
**Приоритет:** P1

### 3.1 Seed-система уровней (Deterministic Level Generation)

**Описание:**
Любая строка (seed) детерминированно генерирует уровень. Игроки могут делиться сидами, соревноваться в одинаковых условиях.

**Алгоритм:**
```
seed_hash = sha256(seed_string)
level_params = {
  length: 2000 + (seed_hash[0] % 10) * 500,      // 2000–7000
  leak_density: 0.05 + (seed_hash[1] % 20) / 100, // 0.05–0.25
  gate_frequency: 5 + (seed_hash[2] % 10),         // 5–15 сек
  weather: WEATHER_TABLE[seed_hash[3] % 5],
  branches: BRANCH_TABLE[seed_hash[4] % 8],
}
```

**Daily Seed:**
- `daily_seed = sha256(UTC_date + "axonos_boundary")`
- Отображается на экране меню: «Daily Challenge: <date>»
- Сообщество может соревноваться в одних условиях без сервера

**Технические требования:**
- SHA-256 — Web Crypto API или wasm-реализация из Rust core
- Генерация — синхронная, < 10 мс
- Seed вводится в UI, валидируется (длина 1–64 символа)

**Сложность реализации:** Low (2–3 дня)
**Приоритет:** P1

---

### 3.2 Призраки (Ghost Replays)

**Описание:**
Локально сохраняются лучшие забеги (до 10). При повторной попытке — полупрозрачный «призрак» Ari показывает, как проходил уровень раньше.

**Реализация:**
- Хранение: `localStorage` под ключом `boundary_run_ghosts`
- Формат: массив `{ seed, proof_hash, inputs: [(time, action), ...], final_score }`
- Размер одного ghost: ~5 KB (1000 inputs × 8 bytes)
- Лимит: 10 ghosts, LRU eviction
- Рендеринг: отдельный полупрозрачный canvas поверх основного, `globalAlpha = 0.3`

**Образовательная метафора:**
Призрак = цифровой след. Ты видишь, как проходил «прошлый ты». Это не конкурент, это напоминание о накопленном опыте.

**Технические требования:**
- Ghost replay использует тот же deterministic engine, что и обычный забег
- Нет отдельного кода для replay — тот же `run_simulation(seed, inputs)`
- Возможность отключить в настройках

**Сложность реализации:** Medium (4–5 дней)
**Приоритет:** P1

---

### 3.3 Deterministic Replay Proof v2 (Enhanced Proof)

**Описание:**
Расширить replay proof, чтобы он включал новые механики.

**Структура proof:**
```json
{
  "version": 2,
  "seed": "user_input_or_daily",
  "config": {
    "contracts": ["C-01", "C-04"],
    "weather": ["W-02"]
  },
  "inputs": [
    { "t": 120, "a": "move_left" },
    { "t": 450, "a": "audit_gate" },
    { "t": 890, "a": "branch_choice", "data": "safe" }
  ],
  "result": {
    "shards": 1240,
    "integrity": 0.87,
    "time_ms": 45230,
    "corruption_level": 1
  },
  "proof_hash": "sha256(seed + config + inputs + result)"
}
```

**Верификация:**
- Любой может взять proof, запустить `verify(proof)` — engine воспроизводит забег и сравнивает result
- Время верификации: < 1 сек на современном браузере
- Возможность «подписи»: `signed_hash = sha256(proof_hash + user_nickname)`

**Технические требования:**
- Обратная совместимость: proof v1 должен верифицироваться как v2 с пустым config
- Верификация — отдельный WASM-модуль, не требует canvas
- Экспорт proof: JSON или QR-код (data URI)

**Сложность реализации:** Medium (5–7 дней)
**Приоритет:** P0 (критично, блокирует остальные фичи)

---

### 3.4 Offline-first PWA (Progressive Web App)

**Описание:**
Игра работает полностью offline после первой загрузки. Service Worker кэширует все assets.

**Требования:**
- `manifest.json` с иконками, theme color, display: standalone
- Service Worker: Cache-first strategy для всех assets
- Размер кэша: < 5 MB (включая все спрайты и WASM)
- Работает в airplane mode

**Образовательная метафора:**
Zero-backend = zero dependency. Игра не требует сети, потому что приватность не должна зависеть от подключения.

**Сложность реализации:** Low (1–2 дня)
**Приоритет:** P2

---

### 3.5 Accessibility Mode

**Описание:**
Полная играбельность без зрения или с ограниченным зрением.

**Фичи:**
| Мод | Описание |
|---|---|
| High Contrast | Все элементы — чёрно-белые или с контрастом > 7:1 |
| Audio Mode | Каждое событие — уникальный звуковой сигнал: утечка (высокий писк), ворота (низкий гул), seal (щелчок), ошибка (диссонанс) |
| Screen Reader | ARIA-live регионы для счёта, статуса, предупреждений |
| Reduced Motion | Отключение trail, shake, glitch-эффектов |
| Large UI | Масштабирование всех UI-элементов ×1.5 |

**Технические требования:**
- Audio: Web Audio API, synthesized tones (не внешние файлы)
- ARIA: `role="status"`, `aria-live="polite"` для game events
- Настройки хранятся в `localStorage` (не персональные данные)

**Сложность реализации:** Medium (4–6 дней)
**Приоритет:** P2

## КРИТЕРИИ ПРИЕМКИ

1. **Determinism:** Любой replay proof v2 верифицируется с точностью до пикселя на любом браузере.
2. **Zero-backend:** Ни одна фича не требует сетевого запроса после первичной загрузки.
3. **Zero-telemetry:** Нет localStorage/cookies, кроме ghosts и настроек accessibility.
4. **Educational integrity:** Каждая новая механика имеет прямую метафору в AxonOS architecture.
5. **Performance:** 60 FPS на Chrome/Firefox/Safari, mobile 30+ FPS.
6. **Bundle size:** WASM + JS + assets < 2 MB gzipped.
